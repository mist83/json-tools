WASM, Privacy, And Why This Library Matters

How this works in plain English

The idea is simple: instead of uploading your JSON to someone else's server, the browser downloads a small app, loads the .NET runtime plus your library, and does the work on the user's machine. The CDN only serves the app files. The JSON never has to leave the tab.

What the user trusts

They trust a static site more than a mystery backend because the private-data path can be designed so there is no API to call in the first place.

What WASM changes

It moves the computation from "their file goes to your server" to "your code goes to their browser."

What stays true

The browser is still sandboxed. This is not a secret local daemon. It is just code running inside the page.

The flow

flowchart LR A["CloudFront / static host"] --> B["Browser downloads app + .NET runtime + JsonUtilities DLL"] B --> C["User picks a local JSON file"] C --> D["Browser streams bytes into scanner / indexer"] D --> E["Only offsets, hashes, and index entries are kept"] E --> F["UI shows matches, byte ranges, and search results"] F --> G["Original JSON stays on the user's machine"]

Why this library is efficient already

  • It works with raw bytes and byte offsets instead of fully turning huge JSON files into giant object graphs.
  • It reuses buffers with ArrayPool<byte> instead of constantly allocating fresh arrays.
  • It can extract only the matching object later, which means "find first, read exact object second."
  • Its semantic index stores byte offsets, not full copied objects.

What that means to a lay person

Imagine looking up books in a library by shelf number instead of photocopying every page into your backpack. This library is trying to remember where things are rather than carrying all the content around all the time.

Important honesty check

The current scanner in this repo still reads the full input stream into a pooled buffer before scanning it. So it is efficient compared with naive JSON parsing, but it is not yet a true constant-memory streamer.

Client-side WASM in one sentence

CloudFront serves code. The browser runs the code. The user's JSON stays local unless the app deliberately sends it somewhere.

What you gain if you ship the DLL in WASM

Trust

The best privacy feature is not collecting the file at all.

Lower infra cost

You are serving static files instead of paying for CPU time every time a curious person tries the tool.

Consistency

The same core library can power the hosted API, localhost mode, and browser-only mode.

Faster feedback

For small and medium files, the user gets immediate results without waiting on uploads.

If you ported it to C or Rust instead

Choice What gets better What gets worse Best fit
C# in WASM Fastest path to shipping because you reuse the current library and mental model. Good safety. Strong maintainability. Bigger runtime payload than a tiny Rust or C WASM module. Garbage collection exists. Raw hot-loop speed usually will not beat a tuned native-style parser. Best choice if time-to-product and code reuse matter most.
C in WASM Potentially the leanest runtime overhead and strongest raw control over memory layout, parser design, and cache behavior. Highest fragility. Hardest to maintain safely. Easiest place to introduce memory bugs or footguns while chasing speed. Best when maximum low-level performance matters more than developer velocity.
Rust in WASM Often close to C-like performance, with much stronger safety. Great fit for streaming parsers, zero-copy-ish slices, and explicit memory discipline. Still a rewrite. More complex than staying in C#. Browser interop and team ergonomics depend on Rust familiarity. Best long-term choice if you want serious performance work without giving up memory safety.

Thought experiment: truly streaming it

Yes, the idea makes sense. In the ideal version, you read a chunk, scan it, keep a tiny amount of carry-over state, emit offsets, and move on. Memory stays roughly flat while file size grows.

But one important correction

The index is usually smaller than the original file, but not guaranteed to be. If you index a huge number of unique terms, n-grams, or repeated offset lists, index overhead can grow a lot. It is safer to say: the index is often a fraction of the file, not always.

What "infinite file" really means

Not truly infinite. It means "far larger than RAM" by using chunked scanning plus a bounded working set. The real limits become browser memory, index size, storage quota, and patience.

The smartest next engineering move

Phase 1

Ship the current C# library into a standalone WASM app so the privacy story becomes true without a rewrite.

Phase 2

Refactor the scanner so it can process chunk-by-chunk instead of reading the whole stream first.

Phase 3

If performance still matters enough, benchmark a Rust parser/indexer core before considering a full rewrite.

Bottom line

The real win from WASM is not magic speed. The real win is that you can hand the user a tool where the computation happens on their machine, the raw data never has to hit your server, and the same core logic can still power the "real" hosted product.
